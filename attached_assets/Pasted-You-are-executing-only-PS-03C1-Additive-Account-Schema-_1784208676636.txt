# Authoritative Account rules

## Account identifiers

Each Account has:

1. A global immutable code:

   ACC-000001

2. A per-game immutable display identifier:

   GOW-001

Both identifiers must:

- be allocated once
- never be changed
- never be reused, including after deletion
- use concurrency-safe allocation
- never use MAX + 1

Do not replace the approved formats with UUIDs or random public identifiers.

## Effective Account status

The effective statuses are:

- AVAILABLE
- PARTIALLY_SOLD
- SOLD
- INACTIVE

Persisted manual override values are only:

- SOLD
- INACTIVE
- NULL

AVAILABLE and PARTIALLY_SOLD must not be persisted as manual override values.

Priority and derivation:

1. Manual INACTIVE override → INACTIVE
2. Manual SOLD override → SOLD
3. All Capacities FINISHED → SOLD automatically
4. All Capacities AVAILABLE → AVAILABLE
5. Any other mixture containing ASSIGNED or FINISHED → PARTIALLY_SOLD

If automatic SOLD was caused only by every Capacity being FINISHED and one Capacity later leaves FINISHED, the effective Account status must be recalculated.

FINISHED is a Capacity state, not an Account status.

No runtime calculation is implemented in PS-03C1. This stage only creates the required schema foundation.

- no counter reuse after Account deletion
- do not create counters for nonexistent Games
- do not insert rows into the live counter table
