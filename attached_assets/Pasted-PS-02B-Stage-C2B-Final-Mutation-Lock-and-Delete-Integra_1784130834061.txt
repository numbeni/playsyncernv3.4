## Part 1 — Immediate UI mutation lock

Add a synchronous local ref lock inside:

- GameFormModal
- ConfirmDialog

The lock must activate before setState or awaiting a Promise.

Required behavior:

- rapid clicks cannot call onSave or onConfirm twice
- Create, Edit and Status cannot appear successful when a request was blocked
- modal/dialog cannot close through Escape, backdrop, Cancel or Close while pending
- lock remains active during successful close
- lock is released after failure so retry is possible

Do not rely only on mutationKey or Provider-level locks.

## Part 2 — Delete Game integration

Use the existing generated client:

- useDeleteGame
- getListGamesQueryKey

Do not create handwritten fetch calls.

Restore Delete actions in the existing Games UI.

For `accountCount > 0`:

- do not send DELETE
- explain that the Game has Accounts
- recommend changing it to INACTIVE

For `accountCount = 0`:

1. show destructive confirmation
2. prevent rapid duplicate confirmation
3. call DELETE /api/games/:id
4. await API success
5. await Games list refetch
6. close only after synchronization
7. do not remove the Game through local state
8. confirm Refresh does not restore it

For Backend 409:

- keep the dialog open
- show a safe Persian dependency message
- do not remove the Game
- recommend INACTIVE

Handle 404, network and server errors without exposing internal details.
