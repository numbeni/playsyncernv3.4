## 1. Make platform changes transaction-safe

The current platform-change flow performs:

- read Game
- check Account history
- update Game

without a transaction or row lock.

Correct this so an Account cannot be created concurrently between the history check and platform update.

For an actual platform change:

- open a database transaction
- load and lock the Game row using PostgreSQL row-level locking such as `FOR UPDATE`
- return 404 when the Game does not exist or is soft deleted
- check all Account history inside the same transaction
- return 409 when any Account history exists
- update the platform inside the same transaction

A request containing the unchanged platform must remain allowed.

Title-only and status-only updates must continue working normally.

Do not redesign Accounts.

## 2. Make hard deletion transaction-safe

The current hard-delete flow performs the history check and delete separately.

Correct it so the operation is atomic:

- open a database transaction
- lock the Game row
- verify the Game exists and is not soft deleted
- check all Account history in the transaction
- return 409 when Account history exists
- hard-delete only inside the same transaction

As defensive protection, map PostgreSQL foreign-key violation code `23503` during Game deletion to controlled HTTP 409 rather than HTTP 500.

Do not cascade-delete any related record.
