## 1. Close the remaining Game-platform concurrency gap

The Games update route already uses a transaction and row lock, but Account creation may still read the Game platform before entering its creation transaction.

Inspect the Account creation flow.

Correct it so Account creation:

- opens a database transaction before reading the Game
- reads and locks the related Game row inside that same transaction
- confirms the Game still exists and is not soft deleted
- uses the platform from that locked Game row
- creates the Account and automatic Capacity records inside the same transaction
- never uses a platform value read outside the transaction

The locking strategy must prevent Account creation and Game platform change from passing concurrently with inconsistent platform/capacity data.

Do not redesign Accounts or Capacity templates.
Do not add new Account features.
This change is only for preserving the Games platform rule.

Use a consistent lock order to avoid deadlocks:

1. lock Game
2. inspect/create Account data
3. create Capacity records
